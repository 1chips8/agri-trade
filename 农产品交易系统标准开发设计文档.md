# 农产品交易系统标准开发设计文档

## 1. 文档说明

本文档描述"农产品直销交易与订单履约系统"的设计方案，涵盖项目目标、系统角色、功能范围、业务流程、数据库设计、接口设计、权限设计、状态设计、技术架构、中间件设计和部署设计。

> **实现状态说明（2026-06-04）**
>
> 本文档同时包含当前实现和目标设计。除非章节中明确标注为已实现，否则不能仅凭设计描述判断功能已经交付。当前尚未实现的主要能力包括：商品评价、30 分钟未支付自动取消、RabbitMQ 异步通知、商品/统计缓存、Redis 商品级库存锁、日统计任务。`product_review` 与 `daily_statistics` 当前仅有表结构。

---

## 2. 项目概述

| 项 | 内容 |
|----|------|
| 项目名称 | 基于 Spring Boot + Vue 的农产品直销交易与订单履约系统 |
| 项目目标 | 构建支持商家入驻、商品发布审核、消费者浏览下单、模拟支付、订单履约、评价、通知和后台统计的交易平台 |
| 架构风格 | 前后端分离 + 模块化单体 |
| 部署方式 | Docker Compose |

### 设计重点

- 核心业务闭环（从商品发布到订单完成全链路）
- 数据库设计与状态流转约束
- 接口权限控制（游客 + CONSUMER/MERCHANT/ADMIN；当前主要在 Service 层校验）
- Redis 防重复提交；商品缓存和并发库存控制为目标设计
- RabbitMQ 延迟消息（超时取消）与异步通知为目标设计
- Docker Compose 一键部署

### 后端模块划分


每个模块内遵循 `controller → service → mapper → entity` 分层，Dto/Vo/枚举按需定义。

## 5. 系统角色与权限

| 角色 | 标识 | 权限范围 |
|------|------|----------|
| 游客 | GUEST | 注册、登录、浏览商品列表和详情 |
| 消费者 | CONSUMER | 购物车管理、下单支付、确认收货、消息、申请成为商家；评价待实现 |
| 商家 | MERCHANT | 商品发布/上下架、订单发货、商家经营统计；评价回复待实现 |
| 管理员 | ADMIN | 商家审核、商品审核、分类管理、订单查询、平台统计 |

### 数据权限边界

| 角色 | 数据隔离规则 |
|------|-------------|
| 消费者 | 只能操作自己的购物车、订单、消息；评价待实现 |
| 商家 | 只能操作自己店铺的商品、订单；评价回复待实现 |
| 管理员 | 可操作平台级全部数据 |

### 鉴权拦截配置

Sa-Token 拦截器对 `/api/**` 统一检查登录状态，角色与数据权限主要由 Service 层显式校验。以下路径无需登录：

| 公开路径 | 说明 |
|----------|------|
| `/api/auth/register` | 用户注册 |
| `/api/auth/login` | 用户登录 |
| `/api/products/**` | 商品列表与详情 |
| `/api/product-categories` | 分类查询 |
| `/doc.html`、`/v3/api-docs/**` | API 文档 |

| 功能 | 说明 |
|------|------|
| 注册 | 用户名/手机号/昵称/密码，写入 `sys_user`，默认角色 CONSUMER |
| 登录 | Sa-Token 生成 UUID 令牌，30 天有效期；当前未配置 Redis DAO，登录态不持久化到 Redis |
| 获取当前用户 | 从 Sa-Token 会话解析用户信息 |
| 退出登录 | 清除 Sa-Token 会话 |

### 6.2 商家入驻模块

| 功能 | 角色 | 说明 |
|------|------|------|
| 提交入驻申请 | 消费者 | 店铺名称/联系人/电话/产地，状态 PENDING |

| 申请列表 | 管理员 | 查看所有入驻申请 |
| 审核通过 | 管理员 | 申请 → APPROVED，创建 merchant 记录，用户角色 → MERCHANT，发送通知 |
| 审核拒绝 | 管理员 | 申请 → REJECTED，记录拒绝原因，发送通知 |

### 6.3 商品分类模块

| 功能 | 角色 | 说明 |
|------|------|------|
| 新增分类 | 管理员 | 分类名/父级ID/排序/状态 |
| 修改分类 | 管理员 | 更新分类信息 |
| 启用/禁用 | 管理员 | 切换分类 status |
| 查询启用分类 | 公开 | 查询所有 ENABLED 状态的分类 |

| 删除条目 | 删除单个购物车项 |
| 清空购物车 | 删除当前用户全部条目 |

## 7. 核心业务流程

### 7.1 商家入驻流程

```
消费者提交申请
    │
    ▼
merchant_apply (apply_status = PENDING)
    │
    ▼
管理员审核
    ├── 通过 ──▶ apply_status = APPROVED
    │              │
    │              ├── 创建 merchant 记录
    │              ├── 用户角色改为 MERCHANT
    │              └── RabbitMQ → 生成通知
    │
    └── 拒绝 ──▶ apply_status = REJECTED
                   │
                   ├── 记录 reject_reason
                   └── RabbitMQ → 生成通知
```

### 7.2 商品发布审核流程

```
商家发布商品
    │
    ▼
product (audit_status = PENDING, sale_status = OFF_SALE)
    │
    ▼
管理员审核
    ├── 通过 ──▶ audit_status = APPROVED（商家可上架）
    └── 拒绝 ──▶ audit_status = REJECTED
```

### 7.3 下单流程（核心事务）

```
选中购物车项
    │
    ├── 校验：商品存在 + 已上架 + 已审核
    ├── 校验：库存充足
    │
    ├── Redis SETNX order:submit:lock:{userId}    ← 防重复提交
    ├── Redis SETNX product:stock:lock:{productId} ← 并发库存控制
    │
    ├── 按 merchantId 分组 → 每个商家生成一个订单
    │
    ├── 事务开始 @Transactional
    │   ├── stock -= quantity        ← 扣库存
    │   ├── INSERT trade_order        ← 订单主表
    │   ├── INSERT trade_order_item   ← 订单明细
    │   ├── INSERT payment_record     ← 支付记录 (PENDING)
    │   ├── INSERT inventory_log      ← 库存流水
    │   └── DELETE cart_item          ← 清理购物车
    ├── 事务提交
    │
    └── RabbitMQ → 投递延迟消息 (30 分钟超时取消)
```

### 7.4 订单状态流转

```
PENDING_PAYMENT ──支付──▶ WAIT_SHIPMENT ──发货──▶ SHIPPED ──收货──▶ COMPLETED
       │                                                              ▲
       └──取消(用户/超时)──▶ CANCELLED ──────────────────────────────────┘
```

### 7.5 超时取消流程

```
订单创建 → 投递延迟消息 (order.cancel.delay, 30min)
    │
    ▼ (30 分钟后)
消费者消费 order.delay.cancel.queue
    │
    ├── 查询订单状态
    ├── 若 order_status != PENDING_PAYMENT → 忽略（幂等）
    └── 若 order_status == PENDING_PAYMENT
        │
        ├── 事务开始 @Transactional
        │   ├── order_status → CANCELLED
        │   ├── stock += quantity           ← 返还库存
        │   ├── INSERT inventory_log        ← 库存流水
        │   ├── payment_record → CLOSED     ← 关闭支付记录
        │   └── INSERT message_notice       ← 生成通知
        └── 事务提交
```

## 11. Redis 设计

| Key | 类型 | 用途 | 生命周期 |
|-----|------|------|----------|
| `login:token:{token}` | String | Sa-Token 登录会话 | 30 天 |
| `product:hot:list` | String/List | 热门商品缓存 | TTL 过期自动刷新 |
| `product:detail:{productId}` | String | 商品详情缓存 | 修改/审核/上下架时删除 |
| `order:submit:lock:{userId}` | SETNX | 防重复提交 | 下单完成后释放 |
| `product:stock:lock:{productId}` | SETNX | 并发库存控制 | 下单完成后释放 |
| `admin:statistics:overview` | String | 管理后台概览缓存 | 短期 TTL |

### 缓存更新策略

- **商品缓存**: 商品修改、审核、上下架时主动删除 `product:detail:{productId}`
- **热门商品**: 过期后由下次请求重新加载
- **统计缓存**: 短期过期，保证数据相对实时

## 12. RabbitMQ 设计

### 拓扑结构

| 组件 | 名称 | 类型 |
|------|------|------|
| 交换机 | `order.exchange` | Direct |
| 交换机 | `notice.exchange` | Direct |
| 队列 | `order.delay.cancel.queue` | Queue (durable) |
| 队列 | `notice.create.queue` | Queue (durable) |
| 路由键 | `order.cancel.delay` | 绑定 order.exchange → order.delay.cancel.queue |
| 路由键 | `notice.create` | 绑定 notice.exchange → notice.create.queue |

### 消息流

```
订单创建成功 ──▶ order.exchange ──[order.cancel.delay]──▶ order.delay.cancel.queue
                    (延迟 30 分钟)                              │
                                                               ▼
                                                    消费者: 检查状态 → 取消/忽略

支付成功 / 审核完成 / 发货 / 取消 ──▶ notice.exchange ──[notice.create]──▶ notice.create.queue
                                                                              │
                                                                              ▼
                                                                 消费者: 写入 message_notice 表
```

### 幂等保障

- 订单取消消费者消费时先查询订单状态
- 非 `PENDING_PAYMENT` 直接 ACK，不做任何操作
- 消息重复投递不会导致重复取消

## 13. 事务设计

### 需要事务包裹的场景

| 场景 | 事务范围 |
|------|----------|
| 审核通过商家申请 | 更新申请状态 + 创建 merchant + 更新用户角色 |
| 提交订单 | 扣库存 + 创建订单 + 创建明细 + 创建支付记录 + 写库存流水 + 清理购物车 |
| 取消订单 | 改订单状态 + 返还库存 + 写库存流水 + 关闭支付记录 + 生成通知 |
| 模拟支付 | 改支付状态 + 改订单状态 + 记录支付时间 + 生成通知 |
| 确认收货 | 改订单状态 + 记录完成时间 |
| 超时取消消费者 | 同取消订单 |

全部使用 Spring `@Transactional` 注解管理。
